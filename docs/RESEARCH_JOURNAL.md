# Research Journal

研究日志是工作台中记录人工观察、判断和复盘理由的附录层。它把一个
研究结论与当时可见的行情快照、策略候选和记录人绑定起来，方便日后复核。
它不是策略管线、模拟账本或券商接口，也不会因为写入一条日志而产生交易
授权。

## 权限边界

```text
已完成的行情快照 + 当前策略元数据
                 |
                 v
       人工填写的研究日志草稿
                 |
                 v
      服务器绑定所有者和记录人
                 |
                 v
       不可变的本地研究记录
                 |
                 X  不创建订单、不改策略、不改模拟账户、
                    不改变券商权限或实盘门禁
```

日志的 `authority` 固定为 `research_only`。以下操作永远不由研究日志
触发：

- 修改 `config/default.json`、策略实验室候选或活动策略；
- 写入、重放或清空模拟账户、影子账户、券商订单和成交账本；
- 创建订单意图、调用券商适配器、消费人工批次批准或解除 kill switch；
- 把一条人工观点计入样本外验证、沙箱对账或实盘就绪门槛。

`WATCH`、`HOLD`、`CONSIDER_REDUCE` 等文字只是研究记录中的判断，不是
买卖指令，也不代表系统认可该判断。

## 在工作台中使用

完成启动和登录后，打开 **研究** 页面即可看到“研究日志”区域。一次
记录包含：

- `研究日期`：这条判断对应的收盘研究日，使用 `YYYY-MM-DD`；
- `记录类型`：观察、判断、交易复盘、风险、策略或周复盘；
- `关联证券`：一个已配置的证券，留空表示组合或全市场；
- `标题` 和 `笔记`：人工写下事实、假设、反例和后续检查；
- `研究观点`：未记录观点、继续观察、研究增配、维持研究判断、研究减配或
  暂不纳入；
- `确信度`：选择观点后填写 0 至 100 的整数百分比。

记录人和登录用户由服务器会话补入，浏览器不能在请求体中指定其他
所有者、伪造记录人或添加 `execution_authorized` 等字段。提交成功后页面
会显示新记录的编号，并在时间线中按研究周分组。

### 修正记录

日志没有编辑或删除操作。发现错误时，在原记录上选择 **追加修正**，填写
一条新记录；新记录的 `correction_of` 指向原编号，旧记录保持原样。这样既
能看到当时的判断，也能看到何时、为什么改正。直接改写 JSON 文件会使内容
指纹校验失败，读取时安全失败，不应作为正常修正流程。

### 筛选与状态

页面支持按类型、证券、关键词和返回条数筛选。筛选只影响当前显示，不改变
记录。以下状态有不同含义：

- **空数据**：当前用户尚未写入记录，或筛选条件没有匹配项；
- **正在写入**：提交请求尚未完成，按钮会暂时锁定，避免重复写入；
- **不可用/失败**：目录、文件锁或记录校验失败，页面会保留明确的恢复提示，
  不会用空白内容冒充“没有记录”。

时间线的周标题来自 `research_date` 对应周一的 `week_start`。当前版本只
提供查询时的周分组，不会自动生成每日摘要、周报、持仓快照或邮件通知；这些
能力仍属于后续迭代范围。

## 记录内容和证据

每条记录保存一个可重算的 SHA-256 `entry_fingerprint`，并包含两类证据：

| 证据 | 可用时记录 | 不可用时记录 |
|---|---|---|
| 行情快照 | 共同完成交易日和快照指纹 | `available=false`，日期和指纹均为空 |
| 策略状态 | 活动候选（可选）、策略指纹和生命周期状态 | `available=false`，候选、指纹和状态均为空 |

证据是“当时看到什么”的证明，不是对行情正确性或策略收益的保证。快照
来源、回退原因和完整行情质量仍应在 **数据** 和 **行情** 页面复核；策略
候选的验证、批准和生命周期仍应在策略实验室复核。记录中的证据不可替代
这些独立审计。

## 本地存储和完整性

默认根目录为 `state/research_journal`。如需分离不同工作区，可以在配置中把它改为
工作区 `state/` 目录下的其他子目录；系统会拒绝空路径、仓库外路径和 `state/`
本身，确保日志仍受 Git 忽略规则保护：

```json
{
  "research_journal": {
    "root_dir": "state/research_journal"
  }
}
```

每个登录所有者使用规范化身份的 SHA-256 目录，不在路径中保存明文用户名：

```text
state/research_journal/
└── users/<owner-sha256>/
    ├── .owner.lock
    └── entries/
        └── journal_<32-hex-id>.json
```

写入在进程锁和操作系统文件锁内完成，记录以同目录临时文件落盘后原子创建；
同一编号不能被覆盖。读取会拒绝重复 JSON 键、未知或缺失字段、错误的文件名、
所有者绑定、日期、证据结构或内容指纹。单个所有者最多 2,000 条记录，单条
记录最多 64 KiB；标题最多 80 个字符，笔记最多 4,000 个字符，查询最多返回
200 条。达到容量后需要由操作者在工作区外归档旧数据，再决定是否恢复写入，
应用不会自动删除历史。

## HTTP 接口

浏览器通常不需要直接调用接口。需要集成只读研究工具时，可使用以下有界
接口：

```text
GET  /api/research?category=<category>&symbol=<symbol>&q=<text>&limit=<1..200>
POST /api/research/journal
```

`GET` 返回研究页数据和 `journal` 对象，包含筛选条件、总数、匹配数、周分组、
记录和固定权限声明。`POST` 只接受 `research_date`、`category`、`symbol`、
`title`、`note`、`decision`、`confidence` 和 `correction_of`；未知字段、
所有者注入和执行授权字段会被拒绝。启用内测登录时，写入需要已认证会话和
会话绑定的 CSRF 令牌；`--owner-local` 只适用于可信的回环地址所有者。

接口错误、损坏记录和目录不可用都会失败关闭。服务端会返回明确错误，前端
显示“不可用”或“失败”状态，不会把错误吞掉后生成一条伪造的研究记录。

## Git、R2 和备份边界

研究日志是本地运行状态，不是发行包内容，也不属于 GitHub 提交。仓库的
`.gitignore` 通过 `state/*` 忽略整个目录；发布校验也拒绝 `state/` 成员。
不要把日志 JSON、截图中的日志内容或内测导出包提交到公共仓库。

Cloudflare R2 的上传白名单只包含已经校验的 `data/cache/*.csv` 和行情
`manifest.json`。`state/research_journal/`（以及模拟账户、助理历史、内测
用户、券商账本和日志）不会进入 R2 行情快照。存储页显示的容量和 A/B 类
预算因此不包含研究日志对象；R2 连接失败也不会影响本地日志读写。

当前没有把研究日志同步到云端的接口。若需要长期留存，应在停止工作台后
对整个工作区的 `state/research_journal/` 做受控、加密的本地备份，并保留
原始目录权限；不要把它放进 R2 行情命名空间，也不要在备份中混入凭据。恢复
后应先让工作台只读检查指纹，再继续追加记录。

## 相关实现和验证

- `src/ai_trade/research_journal.py`：记录 schema、指纹、锁、容量和查询；
- `src/ai_trade/web/service.py`：绑定行情与策略证据、所有者和记录人；
- `src/ai_trade/web/server.py`：认证、CSRF、查询和请求字段边界；
- `tests/test_research_journal.py`、`tests/test_web_research_journal.py`：
  记录不可变性、修正、过滤、损坏检测和 HTTP 输入边界。

研究日志的存在不改变 AI Trade 的核心结论：历史收益不代表未来结果，日志也
不构成投资建议或实盘授权。
